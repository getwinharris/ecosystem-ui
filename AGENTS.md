# ecosystem-ui Agent Instructions

## Scope

This repository owns the public bapX ecosystem static UI:

- `bapx.in`
- `platform.bapx.in`
- `admin.bapx.in`
- `mediahub.bapx.in`
- `blog.bapx.in`
- `docs.bapx.in`
- `avm.bapx.in`
- ecosystem error pages such as `403`, `404`, and `500`

Do not put bapXvm package internals, MCP server implementation, or customer runtime dashboards here.

`bapx.in` is the public ecosystem root and business entry point. It must not become an AVM-only
or VM-resale landing page.

`platform.bapx.in` is the user-facing dashboard for unified login, billing, subscriptions,
product access, hosted apps, and account-level API/MCP entry points.

`admin.bapx.in` is the cross-domain admin surface for OAuth inventory, product status, blogs,
docs, maps, billing configuration, and ecosystem operations.

`avm.bapx.in` is the install/package surface for the headless `@bapX/vm` direction. The package
itself belongs in the bapXvm repository, not in this UI repository.

`vm.bapx.in` is reserved for generated customer runtime proxy hostnames such as
`ACM<8chrandomid>.vm.bapx.in`; the root `vm.bapx.in` host should not serve a product page.

## Design Rules

Follow `DESIGN.md` before changing layout, palette, spacing, typography, buttons, cards, or public
copy rhythm.

Use the supplied bapX SVG assets from `public/brand/`. The main `bapx.in` landing page must use
the plain bapX logo. Product or subdomain surfaces can show the product prefix before the logo, such
as `platformbapx.`, `adminbapx.`, `docbapx.`, `blogbapx.`, or `avmbapx.`.

Keep UI sparse, engineered, and high-trust:

- Near-black canvas, white/ink CTAs, 8px cards, hairline borders, no decorative blob backgrounds.
- Do not put docs-only tables, internal roadmap grids, or every product detail on the root landing.
- Media Hub client examples belong on `mediahub.bapx.in`, not the root landing.
- Blog categories belong on `blog.bapx.in`; docs guides belong on `docs.bapx.in`.

## JSON Schema and Data Rules

This repo uses JSON as the temporary source of truth until the API/database layer is live:

- `src/data/ecosystem.schema.json` defines the shape.
- `src/data/ecosystem.db.json` stores products, OAuth providers, plans, metrics, blog posts, docs
  sections, open-source projects, and Media Hub clients.
- `src/data/ecosystem.ts` provides typed exports for React views.

When changing menus, products, OAuth providers, docs sections, blog categories, Media Hub clients,
plans, or platform/admin dashboard data, update the JSON first and make views read from it. Do not
duplicate the same content in page-local arrays.

OAuth inventory should move toward Google, GitHub, OpenAI, and Apple for unified login. Do not
reintroduce old upstream auth-positioning copy unless the product decision changes.

## Repository Map

`map.mmd` is generated by `scripts/generate-map.mjs`. It maps hosts, routed pages, JSON data,
schema, products, OAuth providers, blog/docs sections, Media Hub clients, open-source projects,
API/MCP gateways, and the generated AVM runtime hostname pattern.

Before non-trivial UI, routing, JSON, docs, host, menu, product, OAuth, or map changes:

```bash
npm run map:repo
```

Do not edit `map.mmd` by hand. Update `scripts/generate-map.mjs` or `src/data/ecosystem.db.json`,
then regenerate.

## OKF Knowledge Rules

Follow the Open Knowledge Format v0.1 draft for repo and ecosystem knowledge:

- Use markdown files with YAML frontmatter for concept documents.
- Every non-reserved concept markdown file must include a non-empty `type` field.
- Prefer `title`, `description`, `resource`, `tags`, and ISO 8601 `timestamp` fields when useful.
- Use `index.md` for progressive disclosure and directory listings.
- Use `log.md` for chronological update history when a folder needs one.
- Use bundle-relative markdown links for cross-project knowledge references where practical.
- Keep knowledge human-readable, agent-readable, diffable, portable, and tolerant of unknown fields.

## Live Delivery

Any change visible on a public bapX domain must be verified, built, deployed, and checked live.

Minimum checks for visible site work:

```bash
npm test
npm run build
```

`npm test` regenerates `map.mmd` before running TypeScript checks.

After deployment, verify:

```bash
curl -k -sS -o /dev/null -w '%{http_code}\n' https://bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://platform.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://admin.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://avm.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://docs.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://blog.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://mediahub.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://vm.bapx.in/
```

Expected root `vm.bapx.in` behavior is `404` unless a generated `ACM<8chrandomid>.vm.bapx.in`
runtime host is being tested.
