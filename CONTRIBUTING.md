# Contributing to ecosystem-ui

This repository controls public bapX ecosystem UI surfaces. Treat changes here as live-product work,
not as isolated mockups.

## Required Reading

Before changing code, docs, data, navigation, styling, routing, deployment, or public copy, read:

1. `AGENTS.md`
2. `DESIGN.md`
3. `map.mmd`
4. `src/data/ecosystem.schema.json`
5. `src/data/ecosystem.db.json`

For workspace-level context, also read `/root/bapx.in/AGENTS.md` when working on the VPS.

## Issue-First Workflow

Use GitHub issues before non-trivial work.

1. Create or link an issue before changing UI, routing, docs, maps, schema, OAuth, billing, domains, deploy behavior, or architecture.
2. Create a branch tied to that issue, preferably `codex/issue-<number>-<short-topic>`.
3. Keep commits focused on the issue.
4. Include validation evidence in the PR or issue update.

Tiny typo fixes can be direct, but any live-site behavior, navigation, design, schema, or workflow change needs an issue first.

## Design And Product Boundaries

Follow `DESIGN.md` for layout, palette, typography, spacing, cards, buttons, and navigation density.

Keep domain roles separate:

- `bapx.in`: public ecosystem root.
- `platform.bapx.in`: signed-in user dashboard only.
- `admin.bapx.in`: internal bapX operator/admin surface.
- `mediahub.bapx.in`: services and client work.
- `blog.bapx.in`: SEO/content engine.
- `docs.bapx.in`: reference docs.
- `avm.bapx.in`: AVM install/package surface.

Do not turn `bapx.in` into an AVM-only page. Do not show platform as a normal public menu item for non-signed-in visitors.

## JSON And OKF Rules

Use JSON as the source of truth for UI data and future backend shape:

- Update `src/data/ecosystem.db.json` before duplicating data in components.
- Update `src/data/ecosystem.schema.json` when data shape changes.
- Keep documents human-readable, agent-readable, diffable, portable, and linked.
- Follow OKF-style frontmatter for new knowledge documents: every non-reserved concept markdown file needs a non-empty `type`.
- Use `index.md` for directory listings and `log.md` for chronological history.

## Map Workflow

`map.mmd` is generated. Do not edit it by hand.

When changing routes, hosts, pages, data, schema, docs, products, OAuth, billing, admin panels, or maps:

```bash
npm run map:repo
```

`npm test` regenerates `map.mmd` before TypeScript checks.

## Validation

Run at minimum:

```bash
npm test
npm run build
```

For visible public changes, deploy and verify live URLs:

```bash
curl -k -sS -o /dev/null -w '%{http_code}\n' https://bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://bapx.in/contact
curl -k -sS -o /dev/null -w '%{http_code}\n' https://platform.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://admin.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://avm.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://docs.bapx.in/
curl -k -sS -o /dev/null -w '%{http_code}\n' https://blog.bapx.in/

```

Expected root `vm.bapx.in` behavior is `404`.

## Commit And PR Checklist

Before opening a PR or landing work:

- Issue is linked.
- Branch name matches the issue.
- `AGENTS.md`, `DESIGN.md`, OKF rules, and `map.mmd` were read.
- JSON schema/data changed before UI duplication.
- `npm test` passed.
- `npm run build` passed for visible UI changes.
- `map.mmd` was regenerated by script, not hand-edited.
- Live URLs were verified after deployment when public behavior changed.
